Datadog RUM Training

This interactive demo shows how Datadog Real User Monitoring captures user behavior in a single-page app — without any page reloads. Work through the pizza wizard and watch RUM events appear in real time.

Training topics

ViewFragment-based view tracking

Each time this wizard changes location.hash, the RUM SDK records a new View event — no page reload needed. The SDK detects hash changes automatically and starts a fresh view with its own load time, action count, and frustration signal tracking.

In RUM › Explorer, set the event type to Views and group by @view.url_hash to see per-step engagement. Each hash is a named wizard step:

#crust→ #sauce→ #cheese→ #toppings→ #size→ #results

You can also correlate view events with error events on the same hash to understand whether a drop-off was caused by a page problem or the user simply leaving.

source:browser service:datadog-demo-app @type:view
| stats count() by @view.url_hash
FunnelConversion funnel by action

In RUM › Explorer, set the scope to Sessions and switch the visualization to Funnel. Each funnel step can be a View or an Action.

For this wizard, the most meaningful action-based funnel tracks the three key events across the full flow:

Step 1  Action: click on Start Building
Step 2  Action: click on See My Pizza
Step 3  Action: pizza_order_submitted

This matches exactly what Datadog shows — sessions that clicked Start Building, then reached the final step, then successfully fired the custom submission action. The drop-off between steps tells you where users abandoned and how many made it through end-to-end.

To investigate a drop-off step, click into it and review individual sessions — session replays will show whether an error blocked the user or they simply left.

MetricCustom metrics from actions

When a user reaches the results screen, this app fires a custom RUM Action with the full order as context attributes:

DD_RUM.addAction('pizza_order_submitted', {
  pizza_order: {
    crust:    "Hand-Tossed",
    sauce:    "BBQ",
    cheese:   "Mozzarella",
    toppings: "Pepperoni",
    size:     "Large (16 in)"
  }
})

In RUM › Generate Metrics, create a count metric from this action. The metric used in the Results dashboard is:

Event type:  Action
Filter:      @action.target.name:pizza_order_submitted
Metric name: rum.pizza_order.submissions
Group by:    @context.pizza_order.crust
             @context.pizza_order.sauce
             @context.pizza_order.cheese
             @context.pizza_order.toppings
             @context.pizza_order.size
             @geo.country
             @geo.country_iso_code

The @geo.country dimension comes from RUM's automatic geo-enrichment — resolved from the user's IP and attached to every event for free.

The Results tab shows the metric broken down using toplist, timeseries, sunburst, and geomap widgets, each filterable via dashboard template variables.

SLOSynthetic tests & error budget

A Datadog Browser Synthetic test runs against this app on a schedule, simulating the full wizard from start to finish. Option order is randomized per run so "always click first" produces varied data.

On ~60% of runs, a simulated JS error is injected at a random step, blocking completion and producing intentional funnel drop-off for training purposes.

An SLO tracks completion rate over a rolling window:

SLO type:   Monitor-based
Target:     99% uptime (1% error budget)
Window:     7-day rolling

The Results dashboard includes an SLO widget. A breach beyond the simulated noise level signals a real problem — investigate in the RUM funnel and session replays.
